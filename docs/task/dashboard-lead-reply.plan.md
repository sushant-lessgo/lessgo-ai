## Overview

Add an AI **"Draft reply"** action to the S4a leads-inbox lead-detail pane (`LeadsInbox.tsx`): for a submission with a real message, one gated AI call drafts an on-brand, editable reply grounded in the owning project's `Project.brief`, which the founder copies to clipboard. Draft + copy only — no send infra, no CRM, no new brand store. Costs `LEAD_REPLY = 1` credit, charged **only on a successful draft** (check-then-charge-on-success). Ownership is enforced via `getAccountScope` page-scope, **not** `FormSubmission.userId` (see Security correction).

### Security correction (overrides spec wording — plan-review confirmed)

The spec says gate on `FormSubmission.userId`. Scout found that field is **attacker-controllable** (written from the client body in `/api/forms/submit`; accepted-and-ignored for back-compat). The trustworthy ownership key is `FormSubmission.publishedPageId ∈ getAccountScope(clerkUserId).pageIds` (derived from server-set `PublishedPage.userId`) — exactly how the S4a inbox scopes. A submission with **null `publishedPageId` has no ownership proof and must be refused** (it's in no account's scope); the spec's "null-publishedPageId → light context" degradation therefore applies only to *thin/absent Brief*, never to unowned submissions.

### E2E note (deterministic-QA placement)

The authed Playwright fixture is single-account and does **not** seed `FormSubmission`s (see the scope note at the top of `e2e/dashboard-rollups-inbox.spec.ts` — cross-user e2e there would be vacuous; the project's precedent covers isolation at the scoping seam via Vitest). We follow that precedent: ownership refusal, charge-on-success-only, no-message gate, and light-context fallback are covered by **Vitest route tests** (mocked prisma/auth/aiClient — established pattern: `src/app/api/social/[token]/posts/route.test.ts`) plus a **jsdom component test** for the pane gating. No new Playwright spec; live-UI eyeball is folded into the phase-4 human gate.

## Phase 1 — Pure helpers: message extraction, brand grounding, prompt

All plain modules (no `'use client'`, no prisma, no server-only imports). **Boundary invariant:** only `messageExtraction.ts` is intended for client import (the pane's gate); `brandGrounding.ts`/`prompt.ts` are server-consumed (route) — documented in the module README (N4).

**Steps**
1. `src/lib/leadReply/messageExtraction.ts`:
   - `extractLeadMessage(data: Record<string, string>): { key: string; value: string } | null` — heuristic:
     - (a) first key whose lowercased name contains `message` / `comment` / `note` / `enquiry` / `inquiry` with a non-empty value;
     - (b) else the longest non-empty free-text value whose key is NOT a contact field (`name`/`email`/`phone` + obvious variants) AND NOT on an explicit **non-message key denylist** — `company`, `subject`, `budget`, `website`, `url`, `city`, `address`, `role`, `title` + variants (N2: prevents `company: "Acme Corporation Ltd"` clearing the floor and offering a draft on a no-real-message submission) — and whose value doesn't itself look like an email/phone/URL;
     - (c) else `null`. Trim values; require substance (≥ 2 words or ≥ 8 chars) so `"yes"`-type values don't count.
   - `hasReplyableMessage(data): boolean` — thin wrapper; this single function is the shared gate for BOTH the UI affordance and the route (one source of truth, per spec "no message → no affordance, don't spend a credit").
   - Note in-file: `previewOf` in `LeadsInbox.tsx` picks *contact* fields — this is intentionally the inverse; do not merge them.
2. `src/lib/leadReply/brandGrounding.ts` — **(B1 fix: derive mode from FACTS, not summary/parse)**:
   - `resolveReplyGrounding(brief: unknown, siteName: string | null): { mode: 'brief' | 'light'; summary: string }`.
   - Reality constraints this design encodes: `BriefSchema` fields are ALL optional → any non-null object parses OK (a thin `{}` still "succeeds"); and `summarizeBrandContext` (`src/modules/email/brandContext.ts` L105-108) NEVER returns empty — thin Brief → fallback sentence "No specific brand facts were captured…". So neither parse-success nor summary-emptiness can detect thinness — both are dead signals.
   - Logic: `BriefSchema.safeParse(brief)`; on parse success → `ctx = buildBrandContext(parsed.data)`; **`mode = 'brief'` IFF `ctx` yielded ≥ 1 real fact** (`ctx.offer || ctx.offerings.length || ctx.audiences.length || ctx.testimonials.length || ctx.proofAvailable.length` — align names to the actual `EmailBrandContext` fields at implementation). Otherwise (zero facts, parse failure, or null/undefined/non-object brief) → `mode: 'light'`.
   - `'brief'` summary = `summarizeBrandContext(ctx)`; `'light'` summary = business/site name only — do NOT ship summarize's "no facts captured" fallback sentence into the prompt. Never throws.
3. `src/lib/leadReply/prompt.ts`:
   - `buildLeadReplyPrompt(grounding, leadMessage, leadName?: string)` → single-call prompt: reply in the business's voice, address the lead's actual question, concise, sendable via email/WhatsApp, no invented facts/prices/availability beyond the grounding, no placeholders like `[Your Name]` beyond a simple sign-off, plain text.
   - `LeadReplyOutputSchema = z.object({ reply: z.string().min(1) })` — the Zod contract for `generateRawJson`.
4. `src/lib/leadReply/README.md` (N4, project convention — agent-oriented module README): purpose, key files, the mode-from-facts rationale (B1), and the boundary invariant — `messageExtraction.ts` is the ONLY file safe/intended for import from `'use client'` code (the pane); grounding/prompt are server-side; nothing here may ever be imported into a published renderer.
5. Unit tests (Vitest):
   - `src/lib/leadReply/messageExtraction.test.ts` — named-message key wins; fallback longest free-text; contact-only data (`name`/`email`/`phone`) → null; subscribe-only (`email` only) → null; **denylist case (N2): `{name, email, company: "Acme Corporation Ltd"}` → null**; empty/whitespace → null; substance floor.
   - `src/lib/leadReply/brandGrounding.test.ts` — **(B1-corrected cases)**: rich valid Brief (`facts.entry` offer/audiences/testimonials populated) → `mode:'brief'`, summary contains offer/audience text; `{}` (parses OK, zero facts) → `'light'`; valid-shape Brief with no `facts.entry` → `'light'`; `null`/`undefined` → `'light'`; non-object garbage (string/number) → `'light'`; ALL paths never throw; light summary contains the site name and NOT the "No specific brand facts" fallback sentence. Also asserts `buildLeadReplyPrompt` embeds the lead message + grounding summary, and light-mode prompt contains no brief-only claims.

## Phase 2 — Draft-reply API route + credit/model config

**Steps**
1. `src/lib/creditSystem.ts` (additive only — billing-beta coordination: whoever merges second reconciles; these are two one-line inserts, low collision):
   - `CREDIT_COSTS` block: add `LEAD_REPLY: 1`.
   - `enum UsageEventType`: add `LEAD_REPLY_GENERATION = 'lead_reply_generation'` (DB `eventType` is a plain String → **no migration**).
2. `src/lib/modelConfig.ts`: add `'lead-reply'` to the `Endpoint` union **+ entries in BOTH tier maps** — `MODELS` is `Record<ModelTier, Record<Endpoint, ModelConfig>>` (L18), so `cheap` AND `production` each need a `'lead-reply'` entry (N1). Reuse the `cold-outreach` model choices per tier (same class of short single-call copy).
3. `src/app/api/leads/[id]/draft-reply/route.ts` (new, POST, account-level — mirrors the modern copy-rails shape of `src/app/api/outreach/[token]/regenerate/route.ts`, EXCEPT charging):
   1. Kill-switch: `NEXT_PUBLIC_LEAD_REPLY_DISABLED === 'true'` → **404 `not_found`** (N3 — deliberate choice: matches the outreach route's conceal-when-disabled behavior AND our ownership 404s, so a disabled feature is indistinguishable from a nonexistent route; 503 rejected — "temporarily off" honesty buys nothing here and adds a second status branch. The `NEXT_PUBLIC_` prefix lets phase 3 hide the button so users normally never hit this).
   2. Clerk auth → 401 if none. Wrap handler in `withAIRateLimit`. Responses via `createSecureResponse`.
   3. Load `FormSubmission` by `params.id` — first select **`publishedPageId` ONLY** for the ownership check; fetch `data` in a second select only after ownership passes (defense-in-depth; reviewer-optional, adopted).
   4. **Ownership:** `getAccountScope(clerkUserId)` (`src/lib/dashboard/accountScope.ts`); refuse with **404** (not 403 — no existence oracle) when: submission missing, `publishedPageId` null, or `publishedPageId ∉ scope.pageIds`. This happens BEFORE reading `data` or any brief. No admin god-view. (`assertProjectOwner` not used — it's token-scoped; this is the account-level analog.)
   5. `extractLeadMessage(data)` → null ⇒ 400 `no_replyable_message` (server enforces the same gate as the UI; no AI call, no charge).
   6. Resolve grounding: `PublishedPage` (already scoped) → its project token/`projectId` → `prisma.project.findUnique({ select: { brief, title/name } })` → `resolveReplyGrounding(project?.brief, siteName)`. Any resolution failure degrades to `light` — never errors (spec constraint).
   7. `checkCredits(userId, CREDIT_COSTS.LEAD_REPLY)` → not allowed ⇒ **402** with the insufficient-credits message + `remaining`.
   8. `generateRawJson('lead-reply', prompt, LeadReplyOutputSchema)` — retry once on failure (outreach `regenerateMessage` pattern), then fail with 500 `generation_failed`. **No charge on failure.**
   9. On success: `consumeCredits(userId, UsageEventType.LEAD_REPLY_GENERATION, CREDIT_COSTS.LEAD_REPLY, { submissionId, grounding: mode })`. Consume-failure branch split (matches `deductCredits` L351-360): balance-changed race (`charge_conflict`-style) → recoverable **500**, NOT a 402 buy-wall; genuine `Insufficient credits` from consume → **402**.
   10. Respond `{ reply, grounding: 'brief' | 'light', remaining }`.
4. `src/app/api/leads/[id]/draft-reply/route.test.ts` (Vitest; mock `@clerk/nextjs`, prisma, `getAccountScope`, `aiClient`, credit helpers — mirror `src/app/api/social/[token]/posts/route.test.ts`). Deterministic-QA cases (assert on mock call counts + which branch fired, not just status codes — inert-assertion lesson):
   - unauthenticated → 401, no prisma/AI calls.
   - submission's `publishedPageId` NOT in caller's scope → 404; **AI never called, no brief read, no `data` fetch**.
   - null `publishedPageId` → 404 (ownership refusal, NOT light-context).
   - contact-only `data` (no message) → 400; AI + consume never called.
   - upfront `checkCredits` insufficient → 402; AI never called.
   - AI failure (both attempts) → 500; `consumeCredits` **never called**.
   - success → 200 with reply; `consumeCredits` called exactly once with `LEAD_REPLY_GENERATION`/cost 1.
   - **(B2) generation succeeds, `consumeCredits` returns a charge-conflict/race failure →** recoverable **500**, NOT 402 (no buy-wall on a race); response distinguishable from `generation_failed`.
   - **(B2) generation succeeds, `consumeCredits` returns genuine `Insufficient credits` →** **402**.
   - thin/absent brief → still 200; prompt passed to AI mock contains light-context (site name), not brief claims.
   - kill-switch env set → **404** (N3); no prisma/AI calls.

**Route imports plain modules only** — nothing from `'use client'` files (LeadsInbox etc.).

**Founder-confirm items (surface at merge, additive/low-risk — do NOT block pipeline):** `LEAD_REPLY = 1` price; the two `creditSystem.ts` inserts vs in-flight billing-beta ordering.

## Phase 3 — Lead-detail pane UI: DraftReplyPanel

1. `src/app/dashboard/leads/DraftReplyPanel.tsx` (new, `'use client'`): self-contained panel taking `{ submissionId, data }`:
   - **Gate:** renders nothing when `!hasReplyableMessage(data)` (same shared helper as the route — the ONLY leadReply module a client file may import, per the phase-1 README) or when `NEXT_PUBLIC_LEAD_REPLY_DISABLED === 'true'`.
   - **Idle:** "Draft reply" button (ui-foundation primitives/`AppIcon`, consistent with pane styling).
   - **Loading:** disabled button + spinner/`Drafting…`.
   - **Draft:** editable `<textarea>` seeded with the reply (founder tweaks before sending), a subtle "less on-brand" hint when `grounding === 'light'` (optional, one line), **Copy** button reusing the exact `FieldRow` clipboard pattern from `LeadsInbox.tsx` L186-218 (`navigator.clipboard.writeText`, `content_copy` → `check` icon, 1.2s "Copied"), and **Regenerate** (re-POST; costs another credit — label it, e.g. "Regenerate (1 credit)").

   - **Other errors:** inline error line + the button re-enabled to retry. Failed generation charged nothing (server guarantees it) — copy can say so.
   - State resets when `submissionId` changes (selecting another lead).
2. `src/app/dashboard/leads/LeadsInbox.tsx` (edit, minimal): render `<DraftReplyPanel submissionId={lead.id} data={lead.data} />` at the end of the detail-pane field list (~L168). No changes to `InboxLead`, list, or S4a aggregation/routes (spec Scope OUT).
